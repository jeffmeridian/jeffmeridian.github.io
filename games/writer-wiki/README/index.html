⚙️ Configuration Sections #

The Wizard is configured via a dedicated panel (toggle with the 🧙‍♂️ icon).

1. Project Metadata

  • Model: Select between local models (e.g., qwen2.5:14b) or cloud models (e.g., gpt-4o).
  • Genre & Tone: Define the core genre (e.g., "Sci-Fi / Noir") and pacing style.

2. Structural Directives

  • Pulp Density Protocol: Controls the density of narrative events vs. introspection.
  • Heat Index Protocol: Manages the "temperature" of the scene (conflict, romance, tension).
  • Anti-Looping: Instructions to prevent repetitive phrasing or plot cycles.

3. Humanization Protocol

  • Emotional Suppression: Directs the AI to be subtle with emotions rather than melodramatic.
  • Banned Words: A list of abstract nouns or clichés to strictly avoid.
  • Simile Restriction:Limits the use of "like" or "as" to force stronger metaphorical language.

4. Character Profiles

Define the cast with specific fields:

  • Voice: How they speak.
  • Internal Monologue: Their private thoughts.
  • Super Power: A unique narrative or physical ability that drives the plot.

5. Style Guide

  • Sensory Rotation: Ensures all five senses are engaged cyclically.
  • Visceral Impact: Focus on physical sensations of emotions.
  • Environmental Friction: How characters struggle against their setting.

6. Chapters

Define the skeleton of your story:

  • Synopsis: A brief summary of what happens.
  • Stop Condition: A narrative trigger that ends the chapter (e.g., "Cliffhanger").
  • Beats: The Wizard will first generate 4 narrative beats based on the synopsis, then write prose for each beat.

⚙️ Generation Pipeline

The Wizard follows a strict two-phase process for each chapter to ensure consistency and depth. This pipeline is unique because it forces the AI to plan before writing, preventing aimless prose.

1. Beat Generation (Structuring)

The AI analyzes the Chapter Synopsis, Scene Goal, and Stop Condition. It breaks the chapter into 4 distinct narrative beats. Each beat acts as a mini-outline, guiding the prose generation.

2. Context-Aware Prose Writing

The compiled beats are processed sequentially. For every beat, the AI receives the last 1,000 characters of the previously generated text (including text from previous chapters). This "sliding window" context ensures that the AI remembers immediate events, dialogue, and tone, maintaining narrative continuity across the entire project.

  3. Click Run Wizard.
  4. The Wizard will stream progress logs to the status window.
  5. Upon completion, a new Folder with the Project Name will be created in the Binder, containing the generated chapters.

  • Manager (manager.py): Orchestrates the workflow by chaining file search → transcription.

🧪 Plugin API Testing #

The Writer's Studio includes a robust Playwright-based test suite specifically designed to test the inter-agent JSON-RPC communication bridge (the plugin_host.py MCP Server) without causing unintended side effects (like actually moving emails or editing live documents).

The Test Suite (transcripts/tests/plugin_api_test.spec.ts)

This test suite performs a realistic, integration-level API test by directly spawning the plugin_host.py backend and communicating with it via standard input/output (STDIO), exactly as the AI Manager Agent does in production.

It tests all 22+ plugins, including:

  • Email Actions: fetch_email, read_email, flag_spam, move_email
  • Document Actions: open_document, edit_active_document
  • Build Automations: build_jeff, make_epub_jeff, syncronize_project

How It Works (The MOCK_PLUGINS Architecture)

To make the tests 100% realistic while remaining completely safe, the system uses a bypass flag architecture:

  1. Spawning the Server: The Playwright beforeAll hook uses Node's child_process.spawn to launch python3 plugin_host.py, injecting a special environment variable: MOCK_PLUGINS=true.
  2. Schema Validation: When the test sends a JSON-RPC request (e.g., {"method": "tools/call", "params": {"name": "flag_spam", "arguments": {...}}}), the plugin_host fully processes the request. It parses the plugin's python file, extracts the docstring schema, and maps the JSON arguments to positional command-line arguments.
  3. Safe Interception: Instead of using subprocess.run to execute the underlying AppleScript or Python code (which would modify the user's system), plugin_host.py intercepts the call immediately prior to execution because it detects the MOCK_PLUGINS=true flag.
  4. Mock Response: It safely returns a standardized [MOCK_RESPONSE] string back through the STDIO pipe.

Why This Matters

This approach ensures we are testing the actual server code and argument parsing logic rather than relying on tautological (self-fulfilling) mocks. If a plugin's argument schema changes or if the JSON-RPC implementation breaks, this test suite will fail, providing high confidence in the stability of the Multi-Agent API without risking user data.

1. Agent Communication Technology #

The system employs a Hub-and-Spoke architecture where a central orchestrator manages multiple specialized agents. The communication backbone relies on JSON-RPC 2.0 over Standard Input/Output (Stdio).

Key Components

  • Orchestrator (manager.py): The central hub that initializes and coordinates all other agents. It uses the AgentOrchestrator class to manage the lifecycle of sub-agents.
  • Sub-Agents: Specialized processes such as model-agent (LLM provider), webagent (browser automation), and eleventy-mcp (site management).
  • Protocol: JSON-RPC 2.0. Messages are serialized as JSON objects with id, method, and params fields.
  • Transport: Stdio Pipes. The orchestrator spawns sub-agents as child processes. It writes requests to the child's stdin and reads responses from the child's stdout.
  • Asynchrony: Python's asyncio library is used throughout. AsyncMCPClient handles the non-blocking I/O, ensuring the orchestrator remains responsive while waiting for agent tools to complete.

Communication Flow

  1. Initialization: The orchestrator launches a sub-agent process.
  2. Tool Call: When the orchestrator needs a specific capability (e.g., specific model details), it constructs a JSON-RPC request (e.g., tools/call) and writes it to the agent's input pipe.
  3. Execution: The agent's event loop reads the request, routes it to the appropriate internal function (e.g., list_models), and awaits the result.
  4. Response: The agent writes the result back as a JSON-RPC response to its output pipe, which the orchestrator reads and processes.

2. Agentic Loop & Test Healing #

The "Test Healing" capability is a specific workflow designed to autonomously repair broken Playwright test cases. It is implemented primarily within the AsyncWebAgentService in webagent_mpc.py.

The Healing Process (smart_heal_test)

The process follows a Retrieve-Analyze-Act cycle:

  1. Initial Verification: The agent first runs the target test file using run_playwright_test.

    • If the test passes, the process terminates early.
    • If it fails, the agent captures the Execution Output (console logs, error messages).

  2. Context Gathering:

    • The agent reads the content of the failing Test Code from the file system.
    • This data (Code + Logs) is packaged into a context dictionary for the LLM.

  3. The Agentic Loop (_run_agentic_loop):

    • Role: The LLM is initialized with a HEALER_PROMPT, instructing it to act as an expert test engineer.
    • Tools: The agent is granted access to:
      • fs_write_file: To modify the test code directly.
      • run_playwright_test: To re-run the test and verify fixes.
      • browser_navigate / browser_snapshot: To inspect the live page if needed.
    • Iteration:
      • The LLM analyzes the failure.
      • It constructs a tool call (e.g., fs_write_file to fix a CSS selector).
      • The system executes the tool and returning the result to the LLM.
      • The LLM can then choose to re-run the test to confirm the fix.
    • Constraints: The loop is capped at a maximum number of turns (default: 10) to prevent infinite loops if the test cannot be fixed.

Summary

This architecture allows the system to treat test failures not as fatal errors, but as tasks to be solved. By combining direct file access with the ability to run and interpret test results, the agent enters a feedback loop that continues until the test passes or the agent determines it cannot be fixed.

How to Use This Engine #

This JSON structure acts as a systemic constraint. Instead of telling the AI what to write, it tells the AI how to think and what to forbid. By enforcing strict linguistic meters and procedural expertise, the resulting output feels less like a generated story and more like a technical log of a high-stakes crisis.

Property & Value Reference Guide #

Linguistic Meter

Value: Short lines, sharp nouns, verbs that move. Average sentence length 8–14 words.

  • Example: "The winch groaned. Naomi kicked the pawl into place. Steel cable hissed under the strain."
  • Explanation: This prevents "AI wordiness" and rhythmic monotony. Short sentences build tension, while the limit on long sentences ensures they are only used to describe complex actions or peak pressure.

The Rule of Rust

Value: Every object shows wear. Any tool described must be used, fail, or become an obstacle.

  • Example: Naomi notices a frayed nylon line while docking; later, that same line snaps when the tug creates a wake.
  • Explanation: This enforces "Object Permanence." If an AI describes a detail for atmosphere, that detail must eventually impact the plot mechanically.

Integrated Expertise

Value: Mandatory VHF etiquette (Over, Out, Roger) and maritime technical accuracy.

  • Example: "Mayday, Mayday, Mayday. This is Vessel Kestrel. Coordinates 41-24 North. Standing by on 16. Over."
  • Explanation: This creates authority in the prose. It replaces generic "phone-style" dialogue with authentic professional communication used in the harbor industry.

Banned Word List

Value: Specific exclusion of "AI-isms" like tapestry, shimmer, and salt air.

  • Example: Instead of "the salt air filled her lungs," the AI writes "the scent of low-tide rot and burnt diesel hit her throat."
  • Explanation: This forces the AI out of its training-data ruts. By banning clichés, the model is forced to generate more specific, gritty, and industrial sensory details.

Injury Cooldown

Value: Persistent physical penalties that do not heal between scenes.

  • Example: Naomi takes a rope burn in Chapter 4; in Chapter 5, her grip strength is compromised while she tries to climb a container.
  • Explanation: This eliminates the "invincible hero" trope. Actions have lasting physical costs that change how the character must solve future problems.

Anti-Looping Protocol

Value: Prohibits repeating plans or recapping internal feelings.

  • Example: Naomi checks a coordinate once; she does not spend three paragraphs wondering if she’s doing the right thing.
  • Explanation: This keeps the narrative "scene-forward." Every paragraph is required to provide new information, a new constraint, or a physical move.

Environmental Friction

Value: Infrastructure and weather act as active antagonists.

  • Example: A slippery algae-covered ramp or a timed security gate becomes a barrier during a chase.
  • Explanation: This grounds the thriller in reality. The harbor itself is a character that provides constant mechanical resistance to the protagonist’s goals.

Pulp Density Protocol

Value: Specific narrative ratios (e.g., 55% Action / 30% Dialogue).

  • Example: Five sentences of Naomi hotwiring a crane for every two sentences of conversation with Inez.
  • Explanation: This calibrates the "density" of the story, ensuring the AI prioritizes movement and physical tasks over static dialogue or "head-space" thinking.

The Internal Pulse

Value: Practical, risk-ledger thinking; fear expressed only through physiological facts.

  • Example: "Risk: High. Battery: 12%. My jaw set so hard the molars ached."
  • Explanation: This defines Naomi’s voice. She doesn't "feel" poetic dread; she calculates the cost of failure and monitors her body's stress responses like a piece of machinery.

Sensory Rotation

Value: Mandatory focus on smell, sound, and touch over sight.

  • Example: The vibration of a diesel engine through the soles of boots or the taste of grit in the teeth.
  • Explanation: This prevents "visual-only" writing. It creates a three-dimensional world by forcing the AI to describe the harbor's unique auditory and tactile landscape.
